## 2. 기술 스택 상세 설명 정리

### 요청 내용

사용자가 `product-plan.md` 섹션 6 "왜 현재 기술 스택인가"에 대해 더 자세한 설명을 요청했다.

### 작업 내용

`daily_reports/docs/tech-stack-report.md`를 기반으로 각 기술 선택의 배경을 다음 7개 항목으로 상세 설명했다.

| 항목 | 핵심 이유 |
|---|---|
| Next.js App Router | SSR(뉴스 상세)과 CSR(홈/개인화)이 한 프로젝트에서 혼재 |
| CORS 프록시 | 개인화/AI/브리핑 서버가 `http://`라 브라우저가 직접 접근 불가 (Mixed Content) |
| Axios + TanStack Query | 4개 서버 응답 포맷이 다르고, 무한스크롤·낙관적 업데이트 내장 지원 |
| NextAuth.js v5 | Google `id_token` → newron `access_token` 2단계 토큰 교환 흐름 |
| Zustand | TTS 오디오 재생 상태 등 서버와 무관한 클라이언트 전역 상태 관리 |
| fetch-event-source | 기본 `EventSource`는 GET만 지원 → 인증 헤더 포함 SSE 불가 |
| Tailwind + CSS Variables | Android `Color.kt` 디자인 토큰을 CSS 변수로 그대로 연결 |

## 3. `product-plan.md` 섹션 6 대폭 확장

### 변경 내용

#### 3-1. 기술별 상세 설명 추가 (6-1 ~ 6-7)

기존 단순 표 하나였던 섹션 6을 다음 구조로 확장했다.

- **6-1** Next.js App Router — 페이지별 SSR/CSR 혼용 이유 (표 포함)
- **6-2** CORS 프록시 — Mixed Content 문제와 `/api/proxy/*` 해결 방식 (코드 예시 포함)
- **6-3** Axios + TanStack Query — 4개 서버 응답 포맷 차이 대응 (포맷 비교 포함)
- **6-4** NextAuth.js v5 — 2단계 토큰 교환 흐름 (`Google OAuth → id_token → newron token`)
- **6-5** Zustand — TanStack Query와의 역할 분리 기준 (`audioStore` 예시)
- **6-6** fetch-event-source — 기본 `EventSource` 한계 보완 이유
- **6-7** Tailwind CSS v4 + CSS Variables — Android 디자인 토큰 연결 방식

#### 3-2. 전체 기술 스택 목록 추가 (6-8)
