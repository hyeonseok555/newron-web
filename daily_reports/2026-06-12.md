## 2. 기술 스택 아키텍처 관점 리서치 문서 작성

### 요청 내용

아키텍처 기획자 관점에서 "왜 이 기술을 썼어요?" 라는 질문에 답할 수 있도록 각 기술 선택의 근거를 대안 비교와 함께 정리.

### 작업 내용

`docs/planning/why-this-stack.md` 신규 생성

### 문서 구조

핵심 제약 6개를 먼저 정의한 뒤, 각 기술을 아래 형식으로 정리:
1. **이 프로젝트의 문제** — 왜 선택이 필요했나
2. **대안과 왜 안 되나** — 다른 선택지의 실패 이유
3. **선택이 이 구조에 맞는 이유** — 코드/다이어그램 포함
4. **한 줄 결론** — 기획자가 바로 말할 수 있는 문장

### 정리된 기술 목록 (11개)

| 기술 | 핵심 선택 이유 |
|---|---|
| Next.js App Router | SSR(뉴스 상세/SEO) + CSR(개인화 피드) 혼재 → 단일 프레임워크로 처리 |
| CORS 프록시 | 3개 HTTP 서버를 브라우저가 직접 호출 불가(Mixed Content) → 서버사이드 중계 |
| Axios | 4개 서버별 인스턴스 + 인증 인터셉터 패턴이 가장 직관적 |
| TanStack Query | 무한스크롤, 낙관적 업데이트, 캐싱을 직접 구현하면 수백 줄 |
| NextAuth.js v5 | Google id_token → newron access_token 2단계 교환을 jwt 콜백이 자연스럽게 수용 |
| Zustand | TTS 오디오 전역 상태 → Context 리렌더 문제 없이 정밀한 구독 제어 |
| fetch-event-source | 기본 EventSource는 GET only + 인증 헤더 불가 → Fetch 기반으로 해결 |
| Tailwind v4 + CSS Variables | Android Color.kt 디자인 토큰을 CSS 변수로 한 번 정의 → 전체 적용 |
| TypeScript | 4개 서버 응답 포맷 차이를 컴파일 단계에서 강제 |
| Biome | ESLint + Prettier를 단일 Rust 도구로 대체 (10~100배 빠름) |
| pnpm | 심볼릭 링크 패키지 관리 → 빠른 설치, 적은 디스크 |
